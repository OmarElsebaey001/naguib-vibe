# Plan 03 — AI Configurator (FastAPI AG-UI Agent Backend)

> Status: `NOT STARTED`

---

## Goal

Build the **FastAPI backend** as an **AG-UI-compliant agent** that communicates with the Next.js console frontend via the AG-UI event protocol over SSE. The agent takes user messages, reads the component catalog and current page config, calls Claude via the Python Anthropic SDK, and emits standardized AG-UI events (text streaming, tool calls, state updates). The AI never generates code — it selects from the catalog and fills content schemas.

### Why AG-UI + FastAPI

- **FastAPI** — async native, `StreamingResponse` for SSE, no timeout constraints (self-hosted), Pydantic for validation
- **AG-UI protocol** — standardized event-driven communication (text streaming, state sync, tool calls, lifecycle)
- **Python Anthropic SDK** — mature, streaming support, structured output
- **Separation of concerns** — frontend (Next.js/Vercel) and AI backend (FastAPI/Railway) scale independently

## Architecture

### Event Flow (Per User Message)

```
User sends message in console
  → Next.js frontend: HttpAgent.runAgent() → POST /api/agent (FastAPI)
  → FastAPI receives request, builds Claude prompt
  → SSE stream begins:
      RUN_STARTED { threadId, runId }
      STEP_STARTED { stepName: "analyzing_intent" }
      ACTIVITY_SNAPSHOT { "Analyzing your request..." }
      STEP_FINISHED { stepName: "analyzing_intent" }
      STEP_STARTED { stepName: "generating_response" }
      TEXT_MESSAGE_START { messageId, role: "assistant" }
      TEXT_MESSAGE_CONTENT { delta: "Sure! I'll..." }  (streamed from Claude)
      TEXT_MESSAGE_CONTENT { delta: "add a pricing..." }
      TEXT_MESSAGE_END { messageId }
      STEP_FINISHED { stepName: "generating_response" }
      STEP_STARTED { stepName: "applying_operations" }
      STATE_DELTA { delta: [JSON Patch operations on page config] }
      STEP_FINISHED { stepName: "applying_operations" }
      RUN_FINISHED { threadId, runId }
  → Frontend applies events: chat updates + preview re-renders
```

### Agent Endpoint

**`POST /api/agent`** — FastAPI route, returns `StreamingResponse` with `text/event-stream` content type.

**Request body** (matches AG-UI `RunAgentInput`):
```python
class RunAgentInput(BaseModel):
    thread_id: str                    # Conversation/project ID
    run_id: str                       # Unique run ID (generated by client)
    messages: list[Message]           # Full conversation history (AG-UI Message format)
    tools: list[Tool]                 # Frontend-defined tools (JSON Schema)
    context: list[ContextItem] = []   # Additional context
    state: PageConfig | None = None   # Current page config state
```

**Response**: SSE stream of AG-UI `BaseEvent` objects (one JSON per `data:` line).

### State Management Strategy

The page config is the shared state between agent and frontend:

1. **`STATE_SNAPSHOT`** — Sent for `replace_all` operations (new page generation). Frontend replaces its entire page config.
2. **`STATE_DELTA`** — Sent for incremental operations. Uses RFC 6902 JSON Patch format via Python's `jsonpatch` library. Frontend applies patches.

### Tool Calls (Frontend-Executed)

The agent can invoke tools that the frontend defines and executes:

| Tool | Purpose | Executed By |
|------|---------|-------------|
| `validate_config` | Validate page config against Zod schemas | Frontend |
| `get_current_config` | Get the current page config state | Frontend |
| `confirm_destructive_change` | Ask user to confirm before replacing entire page | Frontend (shows dialog) |
| `pick_section` | Open section picker UI for user to choose | Frontend (shows picker) |

**Tool call flow is multi-turn**: agent emits `TOOL_CALL_*` events → stream ends (`RUN_FINISHED`) → frontend executes tool (may involve user interaction) → frontend starts a **new** `runAgent()` with the `TOOL_CALL_RESULT` included in messages → agent receives result and continues reasoning.

- [ ] Verify the catalog is small enough to fit in a system prompt (~5-10k tokens)

### 3.2 System Prompt
- [ ] Write the core system prompt:
  - Role definition (landing page configurator, not code generator)
  - Rules (only use catalog types/variants, follow schemas exactly, etc.)
  - The full catalog JSON (injected)
  - Current page config (injected per request via AG-UI state)
  - Available tools it can invoke (validate, confirm, pick)
  - Output format instructions: respond conversationally + emit structured operations
- [ ] Test with Anthropic Python SDK directly before building the endpoint
- [ ] Iterate on prompt until output is consistently valid

### 3.3 AG-UI SDK Integration (Python)
- [ ] Use `ag_ui.core` event models directly — do NOT build a custom emitter class:
  ```python
  from ag_ui.core import (
      RunAgentInput, EventType,
      RunStartedEvent, RunFinishedEvent, RunErrorEvent,
      TextMessageStartEvent, TextMessageContentEvent, TextMessageEndEvent,
      ToolCallStartEvent, ToolCallArgsEvent, ToolCallEndEvent,
      StateSnapshotEvent, StateDeltaEvent,
      StepStartedEvent, StepFinishedEvent,
      ActivitySnapshotEvent,
  )
  from ag_ui.encoder import EventEncoder
  ```
- [ ] Use `EventEncoder` for SSE formatting: `encoder.encode(event)` returns SSE-formatted bytes
- [ ] Use `encoder.get_content_type()` for the `StreamingResponse` media type
- [ ] Use `RunAgentInput` from SDK as the request body model — do NOT redefine it
- [ ] Verify all event types used in Architecture section exist in SDK's `EventType` enum

### 3.4 Operations → JSON Patch Mapping
- [ ] Map each operation type to RFC 6902 JSON Patch operations:
  - `replace_all` → `STATE_SNAPSHOT` with full page config
  - `add_section` → `STATE_DELTA` with `{ "op": "add", "path": "/sections/-", "value": {...} }`
  - `remove_section` → `STATE_DELTA` with `{ "op": "remove", "path": "/sections/{index}" }`
  - `move_section` → `STATE_DELTA` with `{ "op": "move", "from": "/sections/{from}", "path": "/sections/{to}" }`
  - `update_content` → `STATE_DELTA` with `{ "op": "replace", "path": "/sections/{index}/content/...", "value": ... }`
  - `swap_variant` → `STATE_DELTA` with `{ "op": "replace", "path": "/sections/{index}/variant", "value": "..." }`
  - `set_mode` → `STATE_DELTA` with `{ "op": "replace", "path": "/sections/{index}/mode", "value": "dark" }`
  - `set_theme` → `STATE_DELTA` with patches on `/theme/...`
- [ ] Utility: `operations_to_patches(operations, current_config) → list[JsonPatch]`
- [ ] Validate patches produce valid config (apply → validate with Pydantic)

### 3.5 Claude Service (Python Anthropic SDK)
- [ ] `ClaudeService` class wrapping `anthropic.AsyncAnthropic`
- [ ] Method: `async def stream_response(system_prompt, messages, tools) → AsyncIterator`
- [ ] Stream Claude's response token-by-token
- [ ] Parse Claude's structured output (conversational text + operations) from the stream
- [ ] Handle Claude tool_use blocks → map to AG-UI tool call events

### 3.6 Agent Endpoint (FastAPI SSE)
- [ ] `POST /api/agent` route in `routers/agent.py`
- [ ] Use SDK's `RunAgentInput` as FastAPI request body model directly (it's already Pydantic)
- [ ] Extract `thread_id`, `messages`, `tools`, `state` (current page config) from `input_data`
- [ ] Build system prompt with catalog + current config
- [ ] Return `StreamingResponse(event_generator(), media_type="text/event-stream")`
- [ ] `event_generator()` is an async generator that:
  1. Yields `RUN_STARTED`
  2. Yields `STEP_STARTED("analyzing_intent")` + `ACTIVITY_SNAPSHOT`
  3. Calls Claude via `ClaudeService.stream_response()`
  4. As Claude streams text → yields `TEXT_MESSAGE_CONTENT` events
  5. When Claude outputs operations → converts to JSON Patches → yields `STATE_DELTA` or `STATE_SNAPSHOT`
  6. When Claude wants a frontend tool → yields `TOOL_CALL_*` events
  7. Yields `RUN_FINISHED`
  8. On error → yields `RUN_ERROR`

### 3.7 Validation & Retry Loop
- [ ] After computing state patches, apply them to current config and validate with Pydantic
- [ ] If validation fails: retry Claude call with error context (max 2 retries)
- [ ] If retry also fails: yield `RunErrorEvent` with message "Could not generate valid page configuration. Please try a different request." — stop, do not yield invalid patches
- [ ] Yield `ActivitySnapshotEvent` during retry: "Fixing a validation issue, one moment..."

### 3.8 Intent Handling
The AI needs to handle these user intents correctly:
- [ ] "Build me a landing page for X" → `STATE_SNAPSHOT` (full page config)
- [ ] "Add a pricing section" → `STATE_DELTA` (add_section patch)
- [ ] "Remove the FAQ" → `STATE_DELTA` (remove patch)
- [ ] "Swap the hero to a centered layout" → `STATE_DELTA` (swap_variant patch)
- [ ] "Change the headline to ..." → `STATE_DELTA` (update_content patch)
- [ ] "Make the features section dark" → `STATE_DELTA` (set_mode patch)
- [ ] "Use warmer colors" → `STATE_DELTA` (set_theme patches)
- [ ] "Move pricing above the FAQ" → `STATE_DELTA` (move patch)
- [ ] "What hero variants are available?" → text message only, no state events
- [ ] "Undo that" → agent does NOT maintain history; frontend sends previous config via tool result or context. Agent emits `STATE_SNAPSHOT` with the prior config received from frontend

### 3.9 Content Generation Quality
- [ ] AI generates realistic, relevant content (not lorem ipsum)
- [ ] Content matches the user's described business/product
- [ ] Placeholder images use a consistent strategy (placeholder URLs or Unsplash)
- [ ] Icon names are from a defined set (Lucide icon names)
